Full-Stack Observability: Prometheus + Grafana on K8s

Deploy kube-prometheus-stack, write custom ServiceMonitors and recording rules, build actionable Grafana dashboards, and configure Alertmanager — all in a production-ready, GitOps-friendly setup.

PK
Pramod
Cloud & DevOps Engineer
| 📅 3 March 2026 | 📖 11 min read

Flying blind in production is not an option. Without metrics, alerts, and dashboards you are always reacting — never anticipating. In this post I'll walk through deploying a full observability stack using kube-prometheus-stack, writing custom ServiceMonitors, defining recording rules to pre-compute expensive queries, and building an actionable Grafana dashboard that surfaces the four golden signals for every service.

Stack Overview #

The kube-prometheus-stack Helm chart bundles everything you need:

  • Prometheus — metrics collection & storage (TSDB)
  • Alertmanager — alert routing, grouping & deduplication
  • Grafana — dashboards and visualisation
  • kube-state-metrics — cluster-level Kubernetes object metrics
  • node-exporter — per-node OS and hardware metrics
  • Prometheus Operator — CRDs to manage Prometheus via Kubernetes YAML
💡

The Prometheus Operator is the key piece — it lets you declare scrape targets as ServiceMonitor and PodMonitor CRDs instead of editing prometheus.yml directly. This makes the entire config GitOps-friendly.

Installation with Helm #

terminal 
# Add the prometheus-community Helm repo
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

# Install into a dedicated namespace
helm upgrade --install kube-prometheus-stack \
  prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --create-namespace \
  --values monitoring-values.yaml \
  --version 58.3.0

Helm Values

monitoring-values.yaml 
prometheus:
  prometheusSpec:
    retention: 15d
    retentionSize: "40GB"
    resources:
      requests:
        cpu: 500m
        memory: 2Gi
      limits:
        memory: 4Gi
    # Pick up ServiceMonitors from ALL namespaces
    serviceMonitorNamespaceSelector: {}
    serviceMonitorSelector: {}
    # Persistent storage for the TSDB
    storageSpec:
      volumeClaimTemplate:
        spec:
          storageClassName: gp3
          accessModes: ["ReadWriteOnce"]
          resources:
            requests:
              storage: 50Gi

alertmanager:
  alertmanagerSpec:
    storage:
      volumeClaimTemplate:
        spec:
          storageClassName: gp3
          resources:
            requests:
              storage: 5Gi

grafana:
  adminPassword: "change-me-in-prod"   # use a secret in real deployments
  persistence:
    enabled: true
    storageClassName: gp3
    size: 10Gi
  sidecar:
    dashboards:
      enabled: true      # auto-loads ConfigMap dashboards
      searchNamespace: ALL

# Disable components you don't need to reduce footprint
kubeEtcd:
  enabled: false         # managed by cloud provider — not accessible
kubeControllerManager:
  enabled: false
kubeScheduler:
  enabled: false
⚠️

Security: Never commit the Grafana admin password to Git. Use an existingSecret reference or inject it via a sealed secret / external secrets operator.

Custom ServiceMonitor #

Once the stack is running, tell Prometheus how to scrape your own application. Your app must expose a /metrics endpoint in Prometheus text format (use a client library like prom-client for Node.js or the built-in prometheus package for Go/Python).

servicemonitor.yaml 
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: myapp-monitor
  namespace: monitoring          # must match Prometheus operator's namespace or be in scope
  labels:
    release: kube-prometheus-stack   # must match the operator's serviceMonitorSelector
spec:
  namespaceSelector:
    matchNames:
      - production               # the namespace where your app lives
  selector:
    matchLabels:
      app: myapp                 # must match the Service's labels
  endpoints:
    - port: http-metrics         # name of the port in your Service spec
      path: /metrics
      interval: 30s
      scrapeTimeout: 10s
      honorLabels: false
service.yaml (your app's Service) 
apiVersion: v1
kind: Service
metadata:
  name: myapp
  namespace: production
  labels:
    app: myapp
spec:
  selector:
    app: myapp
  ports:
    - name: http
      port: 80
      targetPort: 8080
    - name: http-metrics     # this port name must match the ServiceMonitor
      port: 9090
      targetPort: 9090

Recording Rules #

Recording rules pre-compute expensive or frequently-used PromQL expressions and store the result as a new time series. This keeps dashboards fast and reduces Prometheus CPU load — especially important for high-cardinality metrics.

recording-rules.yaml (PrometheusRule CRD) 
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: myapp-recording-rules
  namespace: monitoring
  labels:
    release: kube-prometheus-stack
spec:
  groups:
    - name: myapp.rules
      interval: 60s
      rules:
        # Request rate (per second, 5-min window)
        - record: job:http_requests_total:rate5m
          expr: |
            sum by (job, namespace, status_code) (
              rate(http_requests_total[5m])
            )

        # Error rate as a ratio
        - record: job:http_error_ratio:rate5m
          expr: |
            sum by (job, namespace) (
              rate(http_requests_total{status_code=~"5.."}[5m])
            )
            /
            sum by (job, namespace) (
              rate(http_requests_total[5m])
            )

        # 99th percentile latency
        - record: job:http_request_duration_seconds:p99_5m
          expr: |
            histogram_quantile(0.99,
              sum by (job, namespace, le) (
                rate(http_request_duration_seconds_bucket[5m])
              )
            )

Alertmanager Configuration #

Alert Rules

alert-rules.yaml 
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: myapp-alerts
  namespace: monitoring
  labels:
    release: kube-prometheus-stack
spec:
  groups:
    - name: myapp.alerts
      rules:
        - alert: HighErrorRate
          expr: job:http_error_ratio:rate5m > 0.05
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "High error rate on {{ $labels.job }}"
            description: >
              Error rate is {{ $value | humanizePercentage }} over the last 5 minutes
              in namespace {{ $labels.namespace }}.

        - alert: HighP99Latency
          expr: job:http_request_duration_seconds:p99_5m > 1.5
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "P99 latency above 1.5s on {{ $labels.job }}"
            description: "P99 = {{ $value | humanizeDuration }}"

        - alert: PodCrashLooping
          expr: |
            increase(kube_pod_container_status_restarts_total[15m]) > 3
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} is crash-looping"

Alertmanager Routing (Slack)

alertmanager-config (in Helm values) 
alertmanager:
  config:
    global:
      resolve_timeout: 5m
    route:
      group_by: ['namespace', 'alertname']
      group_wait: 30s
      group_interval: 5m
      repeat_interval: 3h
      receiver: 'slack-notifications'
      routes:
        - match:
            severity: critical
          receiver: 'slack-critical'
    receivers:
      - name: 'slack-notifications'
        slack_configs:
          - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK'
            channel: '#alerts-warning'
            send_resolved: true
            title: '{{ .GroupLabels.alertname }}'
            text: '{{ range .Alerts }}{{ .Annotations.description }}{{ end }}'
      - name: 'slack-critical'
        slack_configs:
          - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK'
            channel: '#alerts-critical'
            send_resolved: true

Grafana Dashboard via ConfigMap #

With the Grafana sidecar enabled, any ConfigMap labelled grafana_dashboard: "1" in any namespace is automatically loaded as a dashboard — no manual import needed.

grafana-dashboard-configmap.yaml 
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-dashboard
  namespace: production
  labels:
    grafana_dashboard: "1"      # picked up automatically by the sidecar
data:
  myapp-dashboard.json: |
    {
      "title": "MyApp — Golden Signals",
      "uid": "myapp-golden",
      "panels": [
        {
          "title": "Request Rate (req/s)",
          "type": "timeseries",
          "targets": [{
            "expr": "sum(job:http_requests_total:rate5m{namespace='production'})"
          }]
        },
        {
          "title": "Error Rate (%)",
          "type": "timeseries",
          "targets": [{
            "expr": "100 * job:http_error_ratio:rate5m{namespace='production'}"
          }]
        },
        {
          "title": "P99 Latency (s)",
          "type": "timeseries",
          "targets": [{
            "expr": "job:http_request_duration_seconds:p99_5m{namespace='production'}"
          }]
        }
      ]
    }
✅

Tip: Build your dashboard interactively in the Grafana UI first, then export the JSON via Dashboard → Share → Export and paste it into the ConfigMap. This gives you the full schema without writing JSON by hand.

Useful PromQL Quick Reference #

PromQL snippets 
# CPU usage per namespace (top consumers)
sum by (namespace) (
  rate(container_cpu_usage_seconds_total{container!=""}[5m])
)

# Memory usage (working set) per pod
sum by (pod, namespace) (
  container_memory_working_set_bytes{container!=""}
)

# Pods not in Running state
count by (namespace, phase) (
  kube_pod_status_phase{phase!="Running", phase!="Succeeded"}
) > 0

# PVC utilisation (alert when > 85%)
(
  kubelet_volume_stats_used_bytes
  / kubelet_volume_stats_capacity_bytes
) * 100 > 85

# Node disk pressure
kube_node_status_condition{condition="DiskPressure", status="true"} == 1
🚨

High cardinality: Never add unbounded label values (request IDs, user IDs, URLs) to your metrics. Each unique label combination creates a new time series — this can cause Prometheus to OOM in minutes.

Conclusion #

With this setup you get full visibility into your cluster and applications with zero manual Prometheus config editing. Key takeaways:

  • Use ServiceMonitors to declare scrape targets as code — never edit prometheus.yml directly
  • Recording rules keep dashboards fast and reduce query load at scale
  • Ship dashboards as ConfigMaps so they live in Git alongside the app
  • Alert on the four golden signals: latency, traffic, errors, saturation
  • Never put unbounded values in metric labels — cardinality kills Prometheus

Next steps: add Loki for log aggregation and Tempo for distributed tracing to complete the full observability picture — Grafana ties all three together into correlated, drill-down dashboards.
