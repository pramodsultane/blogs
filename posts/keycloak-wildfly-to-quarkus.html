    How I Migrated Keycloak from Wildfly to Quarkus

    A complete migration guide covering every environment variable rename, the new two-phase build model, Dockerfile updates, Helm chart changes, and the lessons we learned moving our production IAM platform to the Quarkus-based Keycloak runtime.

    PS
    Pramod Sultane
    Cloud & DevOps Engineer
    · 15 March 2026 · 15 min read

    If you've been running Keycloak for a while, you've probably heard the news: starting from Keycloak v17, Red Hat dropped the Wildfly-based runtime in favor of Quarkus. This isn't just a version bump — it's a complete rethinking of the container image, its configuration model, and how the server is started.

    Our team at the Technical Platform ran Keycloak on Wildfly for years. It was stable, well-understood, and deeply integrated into our Kubernetes manifests and CI/CD pipelines. But with Wildfly reaching end-of-life support and the Quarkus-based image bringing significant performance and startup improvements, we pulled the trigger on the migration.

    This post documents the lessons we learned, every environment variable that changed (there are many), and how we restructured our Dockerfile and Helm charts to make it work smoothly in production.

    Why Quarkus? The Case for Migrating #

    Wildfly is a full-fledged, heavyweight Java EE application server. It's flexible and battle-tested, but it carries a lot of overhead — slow startup times, high memory consumption, and an XML-heavy configuration model.

    Quarkus, by contrast, is a Kubernetes-native Java framework. It is not an application server. It's designed from the ground up for containerized, cloud-native deployments. The benefits were immediately apparent in our testing:

    ~8s
    Cold start time
    (vs. 60–90s on Wildfly)
    250 MB
    Container image size
    (vs. ~700 MB on Wildfly)
    350 MB
    Idle memory footprint
    (vs. ~800 MB on Wildfly)

    For a production IAM platform running in Kubernetes, these are not small wins.

    MetricWildfly-based KeycloakQuarkus-based Keycloak

    Understanding the New Build Model #

    One of the biggest conceptual shifts in the Quarkus-based Keycloak is the two-phase startup model:

    1. Build phase — Keycloak runs kc.sh build to pre-compile and optimize the server configuration (theme providers, database drivers, etc.). This produces an optimized runtime image.
    2. Start phase — Keycloak runs kc.sh start to launch the server using the pre-built configuration.

    In the Wildfly image, everything was configured at runtime through environment variables. In the Quarkus image, some options are build-time options that must be baked into the image, while others remain runtime options.

    ⚠️

    Important: KC_HTTP_RELATIVE_PATH is a build-time option. If you want to keep the legacy /auth path, this must be set in the Dockerfile during the build step — not in Kubernetes deployment env vars at runtime.

    The recommended production approach is to create a custom Keycloak Docker image that runs the build step at Docker image build time, so containers start immediately without any warm-up penalty:

    Dockerfile 
    # Stage 1: Build the optimized Keycloak image
FROM quay.io/keycloak/keycloak:22.0 AS builder

# Build-time configuration options (baked into the image)
ENV KC_DB=postgres
ENV KC_HTTP_RELATIVE_PATH=/auth

# Copy custom themes
COPY --chown=keycloak:keycloak themes/ /opt/keycloak/themes/

# Run the Keycloak build step
RUN /opt/keycloak/bin/kc.sh build

# Stage 2: Runtime image
FROM quay.io/keycloak/keycloak:22.0

COPY --from=builder /opt/keycloak/ /opt/keycloak/

ENTRYPOINT ["/opt/keycloak/bin/kc.sh"]
CMD ["start"]

    Environment Variable Reference: Complete Change Map #

    This was the most labor-intensive part of the migration. The entire environment variable naming convention changed.

    New Variables (Quarkus-only)

    VariableDescription
    KC_HOSTNAMEThe fully qualified domain name (FQDN) of the Keycloak service. Hostname verification is enabled by default in production. Use --hostname-strict=false only in dev/test.
    KC_HTTP_RELATIVE_PATHSets the path prefix. Defaults to / in Quarkus. Set to /auth for backward compatibility. Build-time option — define in Dockerfile.
    KC_PROXYReplaces PROXY_ADDRESS_FORWARDING. Accepted values: edge, reencrypt, passthrough.
    KC_METRICS_ENABLEDEnables the metrics endpoint. Replaces KEYCLOAK_STATISTICS.
    KC_CACHE_CONFIG_FILEPath to a custom Infinispan cache configuration file for fine-grained control over distributed caching.

    Changed Variables (Renamed from Wildfly)

    Old Variable (Wildfly)New Variable (Quarkus)Notes
    KEYCLOAK_USERKEYCLOAK_ADMINAdmin username
    KEYCLOAK_PASSWORDKEYCLOAK_ADMIN_PASSWORDAdmin password
    DB_VENDORKC_DBe.g., postgres, mysql, mssql
    DB_ADDRKC_DB_URL_HOSTDatabase hostname
    DB_PORTKC_DB_URL_PORTDatabase port
    DB_DATABASEKC_DB_URL_DATABASEDatabase name
    DB_USERKC_DB_USERNAMEDatabase username
    DB_PASSWORDKC_DB_PASSWORDDatabase password
    KEYCLOAK_STATISTICSKC_METRICS_ENABLEDSet to true to enable
    PROXY_ADDRESS_FORWARDINGKC_PROXYSee proxy modes above
    KEYCLOAK_WELCOME_THEMEKC_SPI_THEME_WELCOME_THEMEWelcome page theme name
    JGROUPS_DISCOVERY_PROTOCOLJAVA_OPTSJGroups settings now passed via JVM options
    JGROUPS_DISCOVERY_PROPERTIESJAVA_OPTSMerged into JAVA_OPTS

    Removed Variables (No Direct Equivalent)

    Removed VariableReplacement Strategy
    CACHE_OWNERS_AUTH_SESSIONS_COUNTNo direct equivalent. Use KC_CACHE_CONFIG_FILE to provide a custom Infinispan XML configuration.
    CACHE_OWNERS_COUNTSame as above — requires a custom Infinispan config.

    File and Path Changes #

    Theme Directory

    Update your COPY instructions in Dockerfiles and any volume mount paths in Kubernetes manifests:

    theme path change 
    # Wildfly
/opt/jboss/keycloak/themes/

# Quarkus
/opt/keycloak/themes/

    Configuration File

    The configuration file was also renamed and the format simplified:

    config file path change 
    # Wildfly
conf/keycloak.properties

# Quarkus
conf/keycloak.conf

    The new keycloak.conf uses a simple key=value format. The KC_ prefix is dropped and variable names use hyphens instead of underscores:

    conf/keycloak.conf 
    db=postgres
db-url-host=my-postgres-host
db-url-database=keycloak
http-relative-path=/auth
hostname=keycloak.example.com
proxy=edge
metrics-enabled=true

    Handling the /auth Path Breaking Change #

    The most disruptive change for existing deployments is the default URL path. In Wildfly, Keycloak served everything under /auth. In Quarkus, the default is / (root path).

    Option A — Keep the /auth path (recommended for existing deployments)

    Add the following to your Dockerfile as a build-time argument. This maintains full backward compatibility for all existing clients, redirect URIs, and OIDC discovery endpoints without any client-side changes:

    Dockerfile (keep /auth path) 
    ENV KC_HTTP_RELATIVE_PATH=/auth
RUN /opt/keycloak/bin/kc.sh build

    Option B — Migrate to the new root path

    If you choose to migrate all clients to the new base URL, you must update:

    • OIDC provider configuration in all applications
    • Redirect URIs in every Keycloak client
    • Ingress/proxy rules pointing to /auth
    • Monitoring and health check endpoints
    ✅

    For large deployments with dozens of clients, Option A is strongly recommended as a first step. You can plan a coordinated path migration later once everything is stable on Quarkus.

    Custom Cache Configuration (Replacing CACHE_OWNERS_*) #

    If you relied on CACHE_OWNERS_AUTH_SESSIONS_COUNT or CACHE_OWNERS_COUNT, you now need a custom Infinispan XML file. Mount it into the container and reference it:

    helm-values.yaml (cache config) 
    env:
  - name: KC_CACHE_CONFIG_FILE
    value: /opt/keycloak/conf/cache-ispn-custom.xml
volumeMounts:
  - name: cache-config
    mountPath: /opt/keycloak/conf/cache-ispn-custom.xml
    subPath: cache-ispn-custom.xml
volumes:
  - name: cache-config
    configMap:
      name: keycloak-cache-config
    cache-ispn-custom.xml 
    <?xml version="1.0" encoding="UTF-8"?>
<infinispan
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="urn:infinispan:config:14.0 http://www.infinispan.org/schemas/infinispan-config-14.0.xsd"
  xmlns="urn:infinispan:config:14.0">

  <cache-container name="keycloak">
    <distributed-cache name="authenticationSessions" owners="2">
      <expiration lifespan="-1" />
    </distributed-cache>
    <distributed-cache name="sessions" owners="2">
      <expiration lifespan="-1" />
    </distributed-cache>
  </cache-container>
</infinispan>

    Migration Checklist #

    • Update Docker base image from jboss/keycloak to quay.io/keycloak/keycloak
    • Add kc.sh build step to Dockerfile builder stage
    • Set KC_HTTP_RELATIVE_PATH=/auth as build-time ENV if keeping legacy path
    • Update custom theme COPY paths from /opt/jboss/keycloak/themes/ to /opt/keycloak/themes/
    • Rename KEYCLOAK_USER → KEYCLOAK_ADMIN in all secrets/configs
    • Rename KEYCLOAK_PASSWORD → KEYCLOAK_ADMIN_PASSWORD in all secrets/configs
    • Rename all DB_* variables to KC_DB_* equivalents
    • Replace PROXY_ADDRESS_FORWARDING=true with KC_PROXY=edge
    • Add KC_HOSTNAME set to your FQDN
    • Replace KEYCLOAK_STATISTICS with KC_METRICS_ENABLED=true
    • Migrate JGroups config from JGROUPS_DISCOVERY_* to JAVA_OPTS
    • Replace CACHE_OWNERS_* with custom Infinispan XML via KC_CACHE_CONFIG_FILE
    • Rename conf/keycloak.properties → conf/keycloak.conf
    • Update Ingress/proxy health check paths if needed
    • Update monitoring dashboards & alerting rules for new metrics endpoint format
    • Verify OIDC discovery endpoint is reachable post-migration
    • Run integration tests against all registered clients

    Lessons Learned #

    1. Test the --auto-build path first

    Using CMD ["start", "--auto-build"] in your container is the fastest way to validate that your environment variables are correct before committing to a custom built image.

    2. KC_HTTP_RELATIVE_PATH will bite you if you forget it

    The most common gotcha we encountered was forgetting that this is a build-time option. Setting it only in the Kubernetes deployment env vars does nothing — it must be in the Dockerfile and the kc.sh build must run with it set.

    3. Hostname verification is strict by default

    The new KC_HOSTNAME setting enforces that all requests match the configured hostname. In lower environments, add --hostname-strict=false to the start command to avoid hard-to-debug redirect failures.

    4. Don't skip the two-stage Docker build

    Running kc.sh build inside the same layer that serves traffic works, but it adds ~30 seconds to every container restart. A two-stage Dockerfile eliminates this entirely.

    5. JGroups DNS_PING still works, just configured differently

    The JGroups clustering mechanism that enables Keycloak's distributed session cache still functions — it's just configured via JAVA_OPTS now instead of its own dedicated environment variables.

    Conclusion #

    Migrating from the Wildfly-based Keycloak to the Quarkus-based version requires careful attention to environment variable renames, a new two-phase build model, and path changes throughout your CI/CD configuration. While the change surface is large, it is entirely mechanical once you have the mapping.

    The payoff is substantial: faster startup, smaller images, lower memory usage, and a simpler operational model that aligns with how modern Kubernetes-native applications are built and deployed.

    • Every DB_* variable has a direct KC_DB_* counterpart — just rename
    • The /auth path change is the biggest operational risk; keep it with Option A unless you plan a coordinated client migration
    • The two-stage Docker build pays for itself immediately in restart speed
    • JGroups clustering is alive and well — just lives inside JAVA_OPTS now

    For the full configuration reference, see the official Keycloak All Configurations page.
