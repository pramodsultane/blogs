    Building a RAG-based Chatbot for Your Public Blog

    How to turn your existing blog posts into a conversational assistant using Retrieval-Augmented Generation (RAG), FAISS, Kubernetes, and a lightweight JavaScript chat widget.

    PS
    Pramod Sultane
    Cloud & DevOps Engineer
    · 19 April 2026 · 14 min read

    Most blog chatbots fail for one reason: they answer with generic AI knowledge instead of your actual articles. A Retrieval-Augmented Generation (RAG) pipeline fixes that by grounding every answer in your existing blog posts.

    In this guide, you will build a production-ready chatbot that reads your Markdown content, creates embeddings, stores them in FAISS, exposes an API with FastAPI, and renders a lightweight JavaScript widget directly on your blog.

    💡

    Good technical blog posts do not just explain what to build. They show why, include working code, and end with something readers can copy into their own project within five minutes.

    1. High-level Architecture #

    1.1 Architecture Diagram

    The following diagram shows how user queries flow through the system, from the browser widget to the RAG backend and vector store.

    The chatbot consists of five layers:

    • Content Source – Markdown or MDX blog posts.
    • Ingestion Script – extracts and chunks content.
    • Vector Store – FAISS index containing embeddings.
    • Backend API – retrieves relevant chunks and calls the LLM.
    • Frontend Widget – a small floating chat interface on every page.
    🚀

    The most important design decision is to keep the chatbot focused on your content only. Always include source links in the response so readers can jump back to the original article.

    2. Build the Content Ingestion Pipeline #

    Start by scanning your blog repository for Markdown files. This example works for folders such as posts/, content/, or docs/.

    discover_posts.py 
    from pathlib import Path

blog_files = []

for ext in ["*.md", "*.mdx"]:
    blog_files.extend(Path("content").rglob(ext))

for file in blog_files:
    print(file)

    Once the files are discovered, remove frontmatter and split the content into chunks. Smaller chunks improve retrieval accuracy because the model receives more focused context.

    chunk_posts.py 
    import re
from langchain.text_splitter import RecursiveCharacterTextSplitter

with open("content/build-rag.md", "r", encoding="utf-8") as f:
    raw = f.read()

# Remove YAML frontmatter
clean = re.sub(r"^---.*?---", "", raw, flags=re.S)

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=120
)

chunks = splitter.split_text(clean)
print(f"Generated {len(chunks)} chunks")

    2.1 Create Embeddings and FAISS Index

    After chunking, convert each chunk into an embedding and store it in FAISS.

    build_index.py 
    from sentence_transformers import SentenceTransformer
import faiss
import numpy as np
import pickle

model = SentenceTransformer("all-MiniLM-L6-v2")
embeddings = model.encode(chunks)

index = faiss.IndexFlatL2(len(embeddings[0]))
index.add(np.array(embeddings, dtype="float32"))

faiss.write_index(index, "blog.index")

with open("metadata.pkl", "wb") as f:
    pickle.dump(chunks, f)
    ⚠️

    Do not store only the vector index. Save metadata too, including article title, URL, and the original chunk text. Without metadata, your chatbot cannot cite the source article.

    3. Create the Chat API #

    RAG Query Flow Visualization

    The following interactive demo shows how a single user query flows through embedding, retrieval, and generation steps.

    The backend can be built with FastAPI. On every request it should:

    1. Embed the user question
    2. Search the FAISS index
    3. Build a prompt using the best matching chunks
    4. Return an answer plus source URLs
    app.py 
    from fastapi import FastAPI
from pydantic import BaseModel
import faiss
import pickle

app = FastAPI()

index = faiss.read_index("blog.index")
metadata = pickle.load(open("metadata.pkl", "rb"))

class ChatRequest(BaseModel):
    message: str

@app.post("/api/ask")
def ask(req: ChatRequest):
    # Example retrieval logic
    question_embedding = model.encode([req.message])
    distances, indices = index.search(question_embedding, k=3)

    context = "\n\n".join(metadata[i] for i in indices[0])

    return {
        "content": f"Relevant context found:\n{context[:300]}...",
        "sources": [
            {
                "title": "Building a RAG-based Chatbot",
                "url": "/posts/rag-chatbot"
            }
        ]
    }

    A real implementation would replace the placeholder response with an LLM call using OpenAI, Ollama, or another provider.

    Prompt template example 
    You are an assistant for Pramod's Blog.
Answer only using the provided context.
If the answer is missing, say "I could not find that in the blog posts."

Context:
{{retrieved_chunks}}

Question:
{{user_question}}

    4. Add the Chat Widget #

    The widget should stay minimal. Readers do not want a full-screen chatbot; they want a small helper that answers one question and gets out of the way.

    chat-widget.js 
    async function askQuestion(message) {
  const response = await fetch('/chatbot-api/api/ask', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({ message })
  });

  return response.json();
}

document.getElementById('sendBtn').addEventListener('click', async () => {
  const input = document.getElementById('chatInput');
  const result = await askQuestion(input.value);

  document.getElementById('messages').innerHTML += `
    <div class="message bot">
      ${result.content}
    </div>
  `;
});

    Then include the widget in your blog template:

    Layout integration 
    <script src="/assets/chat-widget.js" defer></script>
<script>
  window.BlogChatWidgetConfig = {
    apiBaseUrl: 'https://blog.example.com/chatbot-api'
  };
</script>

    5. Deploy to Kubernetes #

    Once the chatbot works locally, package it into a container and deploy it.

    deployment.yaml 
    apiVersion: apps/v1
kind: Deployment
metadata:
  name: blog-chatbot
spec:
  replicas: 2
  selector:
    matchLabels:
      app: blog-chatbot
  template:
    metadata:
      labels:
        app: blog-chatbot
    spec:
      containers:
        - name: chatbot
          image: registry.example.com/blog-chatbot:latest
          ports:
            - containerPort: 8000
          env:
            - name: MODEL_NAME
              value: gpt-4.1-mini
            - name: FAISS_INDEX_PATH
              value: /data/blog.index

    Expose the service behind an Ingress such as /chatbot-api so it appears under the same domain as your blog.

    7. Next Steps #

    Once the basic chatbot works, you can improve it further by:

    • Adding source citations under every answer
    • Streaming responses with Server-Sent Events
    • Caching repeated questions in Redis
    • Tracking latency and token cost with Prometheus and Grafana
    • Automatically rebuilding the FAISS index whenever a new article is published
